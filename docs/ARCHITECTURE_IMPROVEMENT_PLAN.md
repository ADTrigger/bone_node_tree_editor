# Bone Node Tree Editor 架构改进方案

## 1. 文档目标

本文档用于整理 `Bone Node Tree Editor` 当前架构存在的主要问题，并给出一套面向后续深度开发的改进方案。

目标分为三类：

- 提升运行时性能，减少无效扫描和不必要的全量同步。
- 降低模块耦合度，为未来增加更多节点类型和复杂编辑能力做准备。
- 建立可持续演进的架构基础，包括稳定的数据绑定、明确的同步策略和更可控的状态管理。

## 2. 当前项目概况

当前插件代码总量约为 1296 行，其中同步相关逻辑集中在 `sync.py` 中，文件规模达到 683 行，已经承担了远超单一模块应负责的职责。

目前主要模块职责大致如下：

- `__init__.py`：插件注册入口。
- `nodes.py`：自定义 `NodeTree` 与 `Node` 定义。
- `operators.py`：手动触发更新与同步的操作符。
- `ui.py`：节点编辑器菜单、头部状态和定时轮询入口。
- `binding.py`：Armature 与 NodeTree 的绑定关系维护。
- `state.py`：全局锁和同步快照缓存。
- `sync.py`：拓扑同步、节点连线修复、选择同步、布局快照和状态仲裁。
- `services.py`：上下文解析、颜色同步和骨骼选择工具方法。
- `layout.py`：节点树自动布局。

当前版本已经具备可用性，但从代码形态上看，整体仍属于功能驱动的 MVP 架构，尚未完成面向长期演进的分层设计。

## 3. 当前架构的核心问题

### 3.1 同步逻辑过于集中

`sync.py` 同时负责以下内容：

- 骨骼与节点拓扑比较
- 选择状态同步
- 节点父子关系编辑回写
- 布局快照记录与恢复
- 链接归一化与冲突清理
- 同步快照更新与状态仲裁

这会带来几个直接问题：

- 任意新需求都容易进入 `sync.py`，导致模块持续膨胀。
- 逻辑职责交织，修改一个同步路径时容易影响另一个路径。
- 难以测试，难以做局部替换或性能优化。

### 3.2 同步机制依赖高频轮询

当前 UI 层通过 `bpy.app.timers` 每 0.2 秒轮询所有 Node Editor，并尝试同步选中状态。

这个设计的问题在于：

- 即使没有任何变化，也会持续执行扫描逻辑。
- 多窗口、多编辑器、大型骨架下，轮询成本会放大。
- 当前定时器主要同步的是“选择状态”，并未形成统一的拓扑同步入口。
- 拓扑更新目前仍依赖手动操作符触发，自动同步链路不完整。

### 3.3 全局状态过于粗粒度

当前 `state.py` 中的 `_node_edit_lock` 和 `_tree_sync_snapshots` 都是全局级别的。

这意味着：

- 一棵树内部的更新锁可能影响另一棵树的行为。
- 多个 Armature 或多个编辑窗口并行工作时，状态边界不清晰。
- 后续增加更多节点树实例、更多编辑模式时，状态冲突风险会明显提高。

### 3.4 快照缓存生命周期不完整

当前快照以 `node_tree.as_pointer()` 作为 key 保存，但清理函数没有接入注册/卸载或对象销毁流程。

潜在风险包括：

- 删除树对象后缓存未清理。
- 插件重载后快照可能残留。
- RNA 指针复用时存在脏快照误命中的风险。

### 3.5 绑定关系依赖名称而不是稳定标识

当前 Armature 和 NodeTree 的绑定主要依靠名称与自定义属性做双向推断。

这会带来以下问题：

- Armature 重命名后可能丢失绑定。
- 复制或派生对象时容易带出过期绑定信息。
- 后续做数据迁移、版本兼容或跨文件管理时不够稳健。

### 3.6 核心逻辑直接依赖 `bpy.context`

当前多个模块直接访问全局 `bpy.context`，部分热路径还使用了 `bpy.ops`。

这会导致：

- 上下文来源不明确，逻辑路径更难推断。
- 调用方和执行环境强耦合。
- 测试几乎无法进行。
- `bpy.ops` 在热路径中通常比直接数据 API 更重，也更容易受到上下文限制。

### 3.7 选择同步缺少明确的“单一真源”

当前节点和骨骼之间都可以改变选择状态，逻辑通过快照对比来判断谁覆盖谁。

这个方案可以工作，但仍然存在问题：

- 冲突仲裁规则不够显式。
- 多入口同时修改时，行为不够稳定。
- 未来如果引入更多交互入口，冲突会更复杂。

### 3.8 布局锁定通过回滚实现

Object 模式下的布局限制目前通过“检测变化后恢复旧布局”实现。

这种方式的问题是：

- 变更已经发生，再做全量恢复。
- 恢复本身也会产生额外写入。
- 性能不理想，也不利于后续扩展更多交互策略。

### 3.9 仓库中已出现未完全接线的接口

例如：

- `bone_node_tree_of`
- `rebuild_tree_from_armature`
- `sync_tree_from_armature`
- `clear_tree_snapshot`
- `clear_all_tree_snapshots`

这说明架构开始出现旁路和中间态接口。若继续堆叠功能而不收敛结构，维护成本会持续上升。

## 4. 这些问题对未来开发的影响

如果当前架构不调整，未来在以下方向上都会越来越吃力：

- 增加新的节点类型，例如约束节点、控制器节点、分组节点、过滤节点。
- 支持批量操作、复杂交互、分层视图和局部刷新。
- 提高大型骨架下的编辑流畅度。
- 增加自动化测试和回归验证。
- 引入持久化配置、版本迁移和数据兼容层。

简单说，当前版本适合继续迭代小功能，但还不适合进入“深度开发阶段”。

## 5. 改进目标

下一阶段架构改造建议围绕以下目标展开：

### 5.1 性能目标

- 尽量避免高频无差别轮询。
- 尽量避免每次同步都全量扫描全部骨骼和节点。
- 将全量重建改为增量 patch 应用。
- 将热路径中的 `bpy.ops` 替换为直接数据 API。

### 5.2 架构目标

- 明确单一真源。
- 将同步、状态、UI、Blender API 适配层拆分。
- 形成可扩展的模块边界。
- 为未来增加更多节点类型和更多同步策略预留空间。

### 5.3 工程目标

- 降低模块之间的隐式依赖。
- 提高代码可测试性和可维护性。
- 建立版本化和迁移能力。
- 让后续协作时更容易定位问题和评估改动范围。

## 6. 推荐的目标架构

### 6.1 单一真源原则

建议明确以下原则：

- Armature 是骨骼业务数据的唯一真源。
- NodeTree 是 Armature 的视图层投影。
- 节点位置、宽度、折叠状态、筛选状态等属于视图状态。
- 父子关系、连接关系、颜色来源、选中骨骼等业务核心状态应以 Armature 为最终依据。

这样做的意义是：

- 节点编辑不再是第二份主数据。
- 所有节点交互都先转换成命令，再修改 Armature，再由 Armature 回放到 NodeTree。
- 同步逻辑会更清晰，冲突仲裁会更稳定。

### 6.2 分层结构建议

建议逐步将代码拆分为以下层次：

```text
bone_node_tree_editor/
  __init__.py
  registration.py
  domain/
    snapshots.py
    diff.py
    commands.py
    policies.py
  application/
    sync_controller.py
    selection_controller.py
    topology_controller.py
    layout_controller.py
  infrastructure/
    binding_store.py
    blender_context.py
    armature_repo.py
    node_tree_repo.py
    event_bridge.py
  presentation/
    nodes.py
    operators.py
    ui.py
```

各层职责建议如下：

- `domain`：纯数据结构、快照、差异、规则和命令，不直接依赖 Blender UI。
- `application`：编排同步流程，决定何时对哪个对象应用什么 patch。
- `infrastructure`：封装 Blender API、数据访问和事件桥接。
- `presentation`：节点定义、菜单、操作符和界面交互。

### 6.3 引入 TreeSession

建议用每棵树独立的 `TreeSession` 替代当前全局状态。

每个 `TreeSession` 可维护：

- `mutation_depth`
- `last_sync_origin`
- `topology_hash`
- `selection_hash`
- `layout_cache`
- `dirty_flags`
- `bound_armature_id`

这样做有几个好处：

- 锁粒度从全局缩小到单树。
- 多树、多窗口时不会互相干扰。
- 生命周期更清晰，更容易清理。
- 后续可以在 session 中挂载性能统计和调试信息。

### 6.4 改为快照差异驱动

建议将同步逻辑改造成：

1. 读取 ArmatureSnapshot
2. 读取 TreeSnapshot
3. 计算差异 diff
4. 生成 patch
5. 应用 patch
6. 更新 session 和缓存

建议的快照类型包括：

- `ArmatureSnapshot`
- `TreeTopologySnapshot`
- `SelectionSnapshot`
- `LayoutSnapshot`

这样可以将同步变成“可推理、可测试、可优化”的流程，而不是一组彼此交错的即时修复逻辑。

### 6.5 从轮询转向事件驱动

建议优先使用 Blender 提供的事件机制，例如：

- `depsgraph_update_post`
- `undo_post`
- `load_post`
- `msgbus`

设计原则建议是：

- 事件驱动作为主同步入口。
- 定时器只作为低频兜底机制。
- 同步目标只处理 dirty tree，而不是扫描所有编辑器。

### 6.6 使用稳定 ID 建立绑定关系

当前绑定方式建议升级为稳定标识方案，例如：

- 为 Armature 和 NodeTree 生成 UUID。
- 在自定义属性中保存双向 ID 绑定。
- 引入 `schema_version` 作为未来升级入口。

这样可以更稳健地处理：

- 重命名
- 复制对象
- 数据迁移
- 旧版本兼容

### 6.7 隔离 Blender 上下文访问

建议把 `bpy.context` 和 `bpy.ops` 封装在基础设施层中。

例如：

- `ContextAdapter`
- `ArmatureRepository`
- `NodeTreeRepository`
- `SelectionService`

业务层不要直接依赖全局上下文，而是通过显式参数和适配层完成交互。

## 7. 性能优化方向

### 7.1 优先减少热路径中的全量扫描

当前最值得优化的点是：

- 所有窗口和所有 Node Editor 的周期性扫描
- 同步过程中对所有 bones 和所有 nodes 的重复遍历
- Object 模式下的整树布局恢复

建议改造方向：

- 只对活动树或 dirty tree 执行同步
- 拓扑变化与选择变化分开处理
- 布局变化只在需要时记录和恢复

### 7.2 拆分拓扑同步与选择同步

拓扑变化和选择变化在频率、成本和处理策略上都不一样，不应放在同一个粗粒度同步流程中。

建议拆为：

- `TopologySyncController`
- `SelectionSyncController`
- `LayoutSyncController`

这样可以实现：

- 选择变化只更新选中状态
- 拓扑变化才触发结构级 patch
- 布局变化只刷新视图状态

### 7.3 尽量避免全量重建

`rebuild_tree_from_armature` 这类全量重建逻辑可以保留，但应作为兜底机制，而不是常规同步路径。

常规情况下应优先：

- 删除缺失节点
- 增量新增节点
- 修复受影响的链接
- 仅同步发生变化的颜色与状态

### 7.4 去掉热路径中的 `bpy.ops`

热路径中调用 `bpy.ops` 往往更慢，也更依赖上下文。

建议：

- 能使用直接数据 API 的地方尽量不用操作符
- 只在用户主动操作、必须借助 Blender operator 时使用 `bpy.ops`

## 8. 推荐的实施路线

### 第一阶段：架构止血

目标是先稳定当前项目，降低继续扩展的风险。

建议任务：

- 将 `_node_edit_lock` 替换为 per-tree session 锁。
- 将快照和缓存迁移到 `TreeSession`。
- 在注册/卸载流程中补齐 session 清理逻辑。
- 将 `sync_tree_from_armature` 纳入正式同步入口。
- 收敛未使用或未接线接口。
- 将热路径中的 `bpy.ops` 尽量改为直接 API。

阶段产出：

- 统一同步入口
- 明确的树级状态边界
- 清晰的缓存生命周期

### 第二阶段：性能重构

目标是从“轮询 + 全量同步”转向“事件 + 增量同步”。

建议任务：

- 引入 dirty flag 机制
- 引入事件桥接层
- 拆分拓扑同步、选择同步、布局同步
- 将全量扫描改为增量 patch 应用
- 降低定时器频率，并将其降级为兜底机制

阶段产出：

- 更低的空闲开销
- 更低的多窗口同步成本
- 更清晰的性能分析边界

### 第三阶段：深度开发基础设施

目标是为未来功能扩展打地基。

建议任务：

- 引入稳定 ID 和 `schema_version`
- 建立迁移逻辑
- 建立基础测试结构
- 引入调试开关和性能日志
- 统一异常恢复策略和降级策略

阶段产出：

- 可迁移的数据结构
- 可追踪的运行行为
- 面向长期维护的工程基础

### 第四阶段：进入功能扩展期

在前三阶段完成后，再推进更复杂的产品能力。

可支持的方向包括：

- 更多节点类型
- 骨骼过滤与分组视图
- 批量编辑
- 约束与驱动关系可视化
- 局部刷新和更复杂的交互能力

## 9. 建议的近期优先级

如果只选择最有价值的几个动作，建议优先顺序如下：

1. 统一同步入口，接入拓扑同步正式流程。
2. 引入 `TreeSession`，替代全局锁和全局快照。
3. 将高频轮询收缩为 dirty tree 同步。
4. 将 `bpy.context` 和 `bpy.ops` 从核心热路径中剥离。
5. 建立稳定 ID 绑定和版本迁移入口。

## 10. 预期收益

完成上述改造后，项目会在以下方面获得明显改善：

- 大型骨架场景下的同步性能更稳定。
- 多窗口、多树并行使用时更可靠。
- 新功能开发不必持续挤进 `sync.py`。
- 逻辑边界更清晰，更容易测试和回归验证。
- 数据层、视图层、交互层职责更明确。
- 为未来的复杂节点系统和深度功能扩展预留足够空间。

## 11. 结论

当前插件的主要问题并不是功能不可用，而是架构还停留在“功能已经跑通，但系统边界还不稳定”的阶段。

如果目标只是继续补几个小功能，当前架构仍然可以支撑一段时间；但如果目标是长期深度开发，那么现在正是合适的重构窗口。

建议优先完成以下三件事：

- 建立统一同步入口
- 引入 per-tree session 状态管理
- 从高频轮询转向事件驱动和增量同步

这三项完成后，插件的性能、稳定性和可扩展性都会有明显提升，也能为后续的复杂节点系统打下更稳固的基础。
